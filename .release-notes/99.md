## Fix Connection header parsing for comma-separated values

The keep-alive decision did exact equality on the whole `Connection`
header value, which meant `Connection: close` worked but
`Connection: close, te` didn't — the latter was treated as the HTTP/1.1
default (keep-alive). The Connection header is a comma-separated list
per RFC 9110 §7.6.1; the presence of `close` (or `keep-alive`) as a
list token is what matters.

After this change, all of the following correctly close the connection:

- `Connection: close`
- `Connection: close, te`
- `Connection: te, close`
- `Connection: Close , te` (OWS / case)
- `Connection: keep-alive, close` (close wins)

This shows up in practice when chained proxies normalize Connection
headers — e.g., HAProxy in front of an origin may emit multi-value
Connection options.
