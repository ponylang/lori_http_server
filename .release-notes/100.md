## Fix Connection header parsing for comma-separated values and multiple header lines

The keep-alive decision did exact equality on the whole `Connection`
header value and only consulted the first matching header line. That
meant `Connection: close` worked but `Connection: close, te` was
treated as the HTTP/1.1 default (keep-alive), and a request with two
field lines (`Connection: keep-alive` followed by `Connection: close`)
silently dropped the second line.

The `Connection` header is a comma-separated list per RFC 9110 §7.6.1,
and per §5.3 multiple `Connection` field lines are equivalent to a
single comma-joined value. The presence of `close` (or `keep-alive`)
as any case-insensitive token, in any `Connection` header, is what
matters.

After this change, all of the following correctly close the connection:

- `Connection: close`
- `Connection: close, te`
- `Connection: te, close`
- `Connection: Close , te` (OWS / case)
- `Connection: keep-alive, close` (close wins)
- Two header lines: `Connection: keep-alive` then `Connection: close`

This shows up in practice when chained proxies normalize Connection
headers — e.g., HAProxy in front of an origin may emit either
multi-value Connection options on a single line or multiple Connection
field lines.
